# ADR-0002 — EUPL-1.2, Bitbucket origin, GitHub public mirror

**Status:** accepted (2026-07-18)

## Decision
License the stack under **EUPL-1.2**. Develop on **Bitbucket** (`precisyn/zbrett`,
private origin); publish a **GitHub public mirror**, also EUPL-1.2, pushed from CI
on `main` and tags (see `bitbucket-pipelines.yml` + `tools/mirror-to-github.sh`).

Licensing is **REUSE-compliant**: `LICENSES/EUPL-1.2.txt`, `REUSE.toml`, and an
SPDX header in every source file. `reuse lint` gates CI.

## Dependency licence compatibility
zBRETT combines only permissive upstreams, all compatible with distributing the
combined work under EUPL-1.2:

| Dependency | Licence | Note |
|---|---|---|
| Reticulum / RNS, LXMF | MIT | permissive |
| libminisketch | MIT | via ctypes |
| Kubo / IPFS | MIT + Apache-2.0 | separate process (HTTP API) |
| zstandard, brotli | BSD / MIT | data compression |
| cbor2, httpx, prometheus-client | MIT/BSD/Apache | permissive |

**RNode firmware is GPLv3** but is a *separate work*: it runs on the board, the
host talks to it over USB-serial (KISS). No linking → no copyleft reach into the
Python stack. Any firmware fork stays in its own GPLv3 repo. EUPL-1.2's §5
compatibility list (GPLv2/v3, AGPLv3, …) is the escape hatch if a copyleft
component ever has to be *linked* in — permissive deps never trigger it.

## Mechanism
Origin is `git@bitbucket.org:precisyn/zbrett.git`; public mirror is
`git@github.com:AlchemicFlare/zbrett.git`. CI adds a `github` remote and pushes
`main` + tags (`tools/mirror-to-github.sh`, run from `bitbucket-pipelines.yml`).

**Auth is an ED25519 deploy key.** GitHub rejects RSA-SHA1, which is all
Bitbucket's Pipelines SSH-keys feature can generate — so the mirror carries its
own key instead: generate an ED25519 pair, register the **public** half as a
**write deploy key** on the GitHub repo, and pass the **private** half
base64-encoded in the secured repo variable `GITHUB_DEPLOY_KEY`. The script
writes it to `~/.ssh/id_mirror` and pushes with `IdentitiesOnly=yes` so no other
key is offered. Local seeding uses your own SSH agent (no variable needed). The
mirror is one-way; issues/PRs stay on Bitbucket/Jira until (if) the project goes
community-facing.
